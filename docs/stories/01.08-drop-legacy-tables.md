# Story 01.08 — Drop tabelas legacy (sprint+1)

## Status
**Draft** — agendar APENAS após 1-2 semanas estáveis pós-cutover (01.07 Done + smoke OK)

## Story
**As a** DBA do Atendechat
**I want** remover as tabelas e colunas que não são mais usadas após o cutover (Baileys, BaileysChats, LidMappings + colunas legacy de Whatsapps)
**So that** o schema fique enxuto e a manutenção futura seja mais simples.

## Acceptance Criteria
1. Migration `YYYYMMDDHHMMSS-drop-baileys-legacy-tables.ts`:
   - `DROP TABLE "Baileys" CASCADE;`
   - `DROP TABLE "BaileysChats" CASCADE;`
   - `DROP TABLE "LidMappings" CASCADE;` (decisão: dropar — uazapi resolve LID sozinha)
2. Migration `YYYYMMDDHHMMSS-drop-legacy-columns-from-whatsapps.ts`:
   - `Whatsapps.session` → DROP COLUMN
   - `Whatsapps.battery` → DROP COLUMN
   - `Whatsapps.plugged` → DROP COLUMN
3. Models `Baileys.ts`, `BaileysChats.ts`, `LidMapping.ts` deletados.
4. `Contact.lid` → **manter** (rastreabilidade; pode voltar a ser útil).
5. Models `Whatsapp.ts`, `Contact.ts` ajustados (remover campos dropados).
6. `database/index.ts` atualizado.
7. Antes da execução: `pg_dump` específico das 3 tabelas exportado para arquivo CSV/SQL no servidor (audit trail).
8. Build limpo + migrations testadas em staging.

## Dev Notes

### Comando de backup (recomendado pelo Carlos antes do PR)
```bash
sudo -u postgres pg_dump --table='public."Baileys"' \
                         --table='public."BaileysChats"' \
                         --table='public."LidMappings"' \
                         atendechat \
                         --data-only \
                         --column-inserts \
                         > /var/backups/atendechat-legacy-dump-$(date +%Y%m%d).sql
gzip /var/backups/atendechat-legacy-dump-*.sql
```

Guardar pelo menos 90 dias.

### Reversão
A migration `down` recria a estrutura mas **não restaura dados** — daí o backup pré-execução. Se for crítico restaurar, usar o dump SQL.
