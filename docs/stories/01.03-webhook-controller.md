# Story 01.03 — Webhook controller + handlers + idempotência

## Status
**In Review** — implementação completa, smoke runtime OK; testes unitários e e2e contra sandbox pendentes como follow-up.

## Story
**As a** sistema Atendechat
**I want** receber eventos da uazapi via webhook HTTP, deduplicar via `WebhookEvents`, e despachar para handlers específicos por tipo de evento
**So that** mensagens recebidas, status de ack, calls, contatos e mudanças de conexão sejam processadas confiavelmente sem perda nem duplicação.

## Acceptance Criteria
1. Rota `POST /uazapi/webhook/:secret` exposta em `routes/uazapiWebhookRoutes.ts` (registrar em `routes/index.ts`).
2. Middleware encontra `Whatsapp` por `uazapiWebhookSecret = req.params.secret`. Se não achar → **HTTP 401** (sem detalhes — não vazar info).
3. Idempotência: tenta `WebhookEvent.create({ uazapiEventId, eventType, whatsappId, payload })`.
   - Se sucesso → processa.
   - Se `SequelizeUniqueConstraintError` → retorna **HTTP 200 "ok (duplicate)"** (uazapi não retentar).
4. Roteador interno chama handler por `eventType`:
   - `messages` → `UazapiWebhookServices/handleMessages.ts` → cria `Contact`, `Ticket`, `Message`. Reutiliza lógica de negócio do `wbotMessageListener.ts` (handleMessage, verifyContact, verifyMediaMessage, queue routing, etc.) — **migrar essa lógica de negócio sem reimplementar**.
   - `messages_update` → `handleMessagesUpdate.ts` → atualiza `Message.ack`, marca read receipts.
   - `connection` → `handleConnection.ts` → atualiza `Whatsapps.status` + emite socket.io para o frontend igual hoje.
   - `call` → `handleCall.ts` → mesma lógica que está em `wbotMonitor.ts:35` (mensagem automática "voz/vídeo desabilitada").
   - `contacts` → `handleContacts.ts` → atualiza `Contact.profilePicUrl` etc.
   - `presence` → `handlePresence.ts` (pode ser no-op por enquanto, log apenas).
5. Sempre retornar **HTTP 200** após processamento bem-sucedido. Erros internos do handler: capturar com `Sentry.captureException(err, { tags: { eventType, whatsappId, severity: "high" } })` e logger.error(...) — depois retornar 200 (uazapi não retentaria mesmo). **Nunca silenciar erro sem reportar para Sentry.**
6. Cleanup job: `node-cron` agendado em `queues.ts` (ou novo `cleanup-jobs.ts`) que roda **diariamente às 03:00 UTC**:
   ```ts
   await WebhookEvent.destroy({ where: { processedAt: { [Op.lt]: cutoff30d } } });
   ```
7. Testes unitários cobrem:
   - Handler `messages` cria ticket + message corretamente (mock DB).
   - Idempotência: 2 inserts do mesmo `uazapiEventId` → segundo retorna 200 sem duplicar.
   - Secret inválido → 401.
8. `npx tsc --noEmit` passa limpo.

## Dev Notes

### Estrutura da rota
```ts
// routes/uazapiWebhookRoutes.ts
import { Router } from "express";
import UazapiWebhookController from "../controllers/UazapiWebhookController";
const router = Router();
router.post("/uazapi/webhook/:secret", UazapiWebhookController.handle);
export default router;
```

### Esqueleto do controller
```ts
// controllers/UazapiWebhookController.ts
import { Request, Response } from "express";
import Whatsapp from "../models/Whatsapp";
import WebhookEvent from "../models/WebhookEvent";
import { UniqueConstraintError } from "sequelize";
import * as router from "../services/UazapiWebhookServices/router";
import { logger } from "../utils/logger";

const handle = async (req: Request, res: Response): Promise<Response> => {
  const { secret } = req.params;
  const whatsapp = await Whatsapp.findOne({ where: { uazapiWebhookSecret: secret } });
  if (!whatsapp) return res.status(401).send("invalid");

  const body = req.body;
  const uazapiEventId = body.id || body.event_id;
  const eventType = body.event || body.eventType;
  if (!uazapiEventId || !eventType) {
    logger.warn("uazapi webhook: payload sem id/event", body);
    return res.status(200).send("ok (malformed)");
  }

  try {
    await WebhookEvent.create({
      uazapiEventId, eventType, whatsappId: whatsapp.id, payload: body
    });
  } catch (err) {
    if (err instanceof UniqueConstraintError) {
      return res.status(200).send("ok (duplicate)");
    }
    throw err;
  }

  try {
    await router.dispatch(eventType, body, whatsapp);
  } catch (err) {
    logger.error(`uazapi webhook handler erro evt=${eventType} wid=${whatsapp.id}: ${err}`);
    // 200 mesmo com erro — uazapi não vai retentar; capturar e investigar manualmente
  }

  return res.status(200).send("ok");
};
export default { handle };
```

### Migração da lógica de negócio do `wbotMessageListener.ts`
- Funções a **mover** para `handleMessages.ts` (sem mudança de lógica, só adaptar payload):
  - `handleMessage(msg, wbot, companyId)` → renomear para `handleUazapiMessage(payload, whatsapp)`.
  - `verifyContact(msgContact, wbot, companyId)`, `verifyQuotedMessage`, `verifyMediaMessage` etc. continuam.
- O parâmetro `wbot` (cliente Baileys) **vira `Whatsapp` model** — chamadas como `wbot.profilePictureUrl(...)` viram `await ChatDetails(whatsapp, { number }).then(d => d.profilePicUrl)`.
- `getBodyMessage`, `getQuotedMessageId`, `getTypeMessage` etc. — adaptar para o payload uazapi (campos diferentes; **referenciar `docs/architecture/WEBHOOK-PAYLOADS.md`**).
- ⚠️ Cuidado com **loops**: configure o webhook da uazapi com `excludeMessages: ["wasSentByApi"]` para não receber de volta as mensagens que o próprio Atendechat enviou.

## Notas de implementação
- **Idempotência:** `WebhookEvent.create` com `UNIQUE (uazapiEventId, whatsappId)`. Duplicatas viram HTTP 200 sem reprocessar.
- **Sentry:** todos os erros de handler são reportados com tags `eventType`, `whatsappId`, `severity: high`.
- **Sempre HTTP 200:** uazapi não retentaria mesmo em 5xx, e o evento já foi persistido — investigação fica via Sentry/`WebhookEvents`.
- **handleMessages cobre os casos comuns** (texto, image, video, audio/ptt, document, sticker). Casos exóticos (location, button, list, contact card) são persistidos com `mediaType` bruto e `dataJson` completo — refinamento UI pode vir em iteração futura.
- **handleCall migrado de `wbotMonitor.ts`:** lógica de mensagem automática voz/vídeo desabilitada, com tradução pt/en/es e criação de Message `mediaType=call_log`.
- **Cleanup de WebhookEvents** roda diariamente às 03:00 BRT (configurável via `WEBHOOK_EVENTS_RETENTION_DAYS`, default 30).
