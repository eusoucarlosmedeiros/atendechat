## Dev Notes

### Pseudocódigo do novo `SendWhatsAppMessage`
```ts
import SendText from "../UazapiServices/send/SendText";
import ShowWhatsAppService from "../WhatsappService/ShowWhatsAppService";

const SendWhatsAppMessage = async ({ body, ticket, quotedMsg }: Request): Promise<Message> => {
  const whatsapp = await ShowWhatsAppService(ticket.whatsappId, ticket.companyId);

  // uazapi aceita number puro (E.164) OU JID. Quando o contato é grupo,
  // mandamos o jid completo. Caso contrário, só o number.
  const number = ticket.isGroup
    ? `${ticket.contact.number}@g.us`
    : ticket.contact.number;

  try {
    const response = await SendText(whatsapp, {
      number,
      text: formatBody(body, ticket.contact),
      replyid: quotedMsg?.id
    });
    await ticket.update({ lastMessage: formatBody(body, ticket.contact) });
    // Persistência da Message é feita posteriormente quando o webhook
    // 'messages' (com from_me: true) chegar — evita race condition.
    // Alternativamente, retornar o ID da uazapi para persistência imediata.
    return response;
  } catch (err) {
    Sentry.captureException(err);
    throw new AppError("ERR_SENDING_WAPP_MSG");
  }
};
```

### Sobre persistência da Message enviada
**Decisão arquitetural:** quando o operador envia, a uazapi vai eventualmente disparar webhook `messages` com `from_me: true` ecoando a mensagem. Isso é a **fonte de verdade**.

Mas o frontend espera ver a mensagem **imediatamente** após o clique (UX). Então:
- **Opção A (preferida):** persistir `Message` síncrono com o `response.id` da uazapi. Quando o webhook chegar, fazer `Message.upsert` (idempotente).
- **Opção B:** persistir somente quando webhook chega, e o frontend fica com mensagem "fantasma" 1-2s. UX pior.

Vou com **Opção A**.

### Tratamento de LID/PN
A uazapi resolve internamente. Logo:
- O `Contact.number` deve ser **PN puro** (E.164) sempre que possível.
- Se `Contact.number` ainda estiver com LID (legado da Baileys), a uazapi pode aceitar e tentar resolver, mas pode falhar.
- **Precaução:** rodar `backfillLidMappings` (script existente) antes do cutover para corrigir contatos legados ao máximo.
